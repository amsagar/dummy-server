## 1. What this project is

A **Spring Boot 4 + Spring AI** backend that hosts a tool-using LLM agent, plus
a **React 19 + Vite 6** admin/chat console. The agent is designed to be
**registry-driven**: every model, tool, skill, MCP server, decision table,
guardrail policy, and agent profile is rows in Postgres and is hot-swappable
without redeploy.

The flagship business use-case is **PODS order validation** — checking a
shipping order's leg sequence, serviceability per leg, and container
availability. That flow is implemented as a **Skill** (`pods-order-validation`)
plus a **DMN decision table** ("Leg Sequences") plus three HTTP tools
(`Get_OrderID`, `Serviceability`, `ContainerAvailability`).

Beyond order validation, the same platform is used as a general-purpose chat
agent (per `application.yaml`'s `vendor-rationalization-ui` client and the
free-form chat UI).

## 5. HTTP API surface

All endpoints are under `/api/v1/`. JWT is HS256 with `sub = userId`, claim
`email`, expiry from `pods.auth.jwt-expiration-ms`.

### 5.1 Auth (`/api/v1/auth/*` — public)

| Method | Path | Purpose |
|---|---|---|
| POST | `/signup` | Create account → `{ token, user }` |
| POST | `/login`  | Login → `{ token, user }` (401 on bad creds) |

### 5.2 Chat & sessions (public — but session ownership is enforced)

When no JWT is present, the request runs as the default user
**`order-validation-ui`**, unless the `X-OV-Client` header is one of the
allow-listed values (`order-validation-ui`, `vendor-rationalization-ui`), in
which case that string becomes the persisted user id. This is enforced in
`SecurityContextService`.

**`POST /api/v1/chat`** — streams SSE (`text/event-stream`) of one turn.
Request body fields: `message`, `attachments`, `sessionId`, `timezone`,
`runtimeMode`, `modelSelectionMode`, `agentProfileId`, `model`/`modelId`,
`embeddingModel`. Event types emitted (each payload is JSON with `type`,
`schemaVersion`, `eventId`, `emittedAt`):

`connected`, `text.delta`, `reasoning.delta`, `plan.created`, `task.started`,
`task.done`, `step.started`, `step.finished`, `tool.match`, `tool.call`,
`tool.done`, `tool.result`, `question`, `approval_required`, `state.updated`,
`cost.updated`, `summary.updated`, `session.updated`, `done`, `error`.

Other chat endpoints:

| Method | Path | Purpose |
|---|---|---|
| POST | `/chat/reply` | HITL answer to `question` |
| POST | `/chat/approve`, `/chat/reject` | HITL `approval_required` answer |
| DELETE | `/chat/messages/{id}` | Delete a single message |
| POST | `/chat/sessions/{id}/cancel` | Cancel an in-flight turn |
| POST | `/chat/sessions/{id}/truncate` | Truncate from a given message |
| GET | `/chat/history/{sessionId}` | Paginated message history |
| GET | `/chat/cost/{sessionId}` | Cost summary + per-turn breakdown |
| GET | `/chat/pending/{sessionId}` | Pending HITL interactions |
| GET | `/chat/events/{sessionId}` | Filtered runtime events |

### 5.3 Sessions (`/api/v1/sessions/*` — public)

CRUD over `chat_sessions`: list (paginated, archived filter), rename, archive,
delete, get metadata, get messages.

### 5.4 Models (`/api/v1/models/*` — public; `/api/v1/providers` is JWT-gated)

CRUD over `supported_models` (chat models). The **registry-key endpoint
`/api/v1/models` POST is public** — anyone reachable can register a model with
an API key. (See §13.)

### 5.5 Embedding models (`/api/v1/embedding-models/*` — JWT)

Same shape as `/models`, plus `setDefault`. Embedding model selection drives
the pgvector index dimension (3072 padded/truncated).

### 5.6 Tool domains & tools (`/api/v1/tool-domains/*` — JWT)

Tool domain CRUD; per-domain tool CRUD; import from **OpenAPI / cURL /
Postman**; per-tool test; install framework defaults. All responses redact
secrets via `publicToolView`.

### 5.7 Tool auth (`/api/v1/tool-auth/*` — JWT)

Per-domain `tool_auth_profiles` CRUD (DB constraint: **one profile per
domain**); per-tool binding (`authProfileId` + `authOverrideEnabled`); OAuth
helpers — `authorization-url`, `oauth/callback`, connect/test/reauthenticate
flows for profiles **and** individual tools.

> **Subtle gotcha:** `HttpToolAuthService` resolves auth by picking the
> domain's single enabled profile, regardless of the tool's `authProfileId`,
> unless `authOverrideEnabled = true` (in which case credentials on the tool
> row are used). The `authProfileId` column is stored and exposed in APIs but
> not actually consulted at request time. See `HttpToolAuthServiceTest` for
> the verified behavior.

### 5.8 Skills (`/api/v1/skills/*` — JWT)

Skill CRUD, file CRUD inside a skill, file tree, multipart upload of
`.md` / `.zip` / `.skill`. Root `SKILL.md` cannot be deleted.

### 5.9 Decision tables (`/api/v1/decision-tables/*` — public)

CRUD + `POST /{name}/evaluate` — evaluates the DMN JSON with provided inputs.
Used by the `decisionTableEvaluate` tool.

### 5.10 MCP registry (`/api/v1/mcp-registry/*` — JWT)

Server registry + discovery + reconnect/reauth. Connect endpoints support
API-key, bearer, basic, client-credentials, and auth-code flows. Both JSON
callback (`POST /{id}/oauth/callback`) and browser callback (`GET
/oauth/callback`). Per-tool enable/disable, bulk enable/disable-all.

### 5.11 Platform admin (`/api/v1/platform/*` — JWT)

- **Guardrail policies** (`/policies`): CRUD over `guardrail_policies`.
- **Agent profiles** (`/agent-profiles`): CRUD over `agent_profiles`.
- **MCP raw repo** (`/mcp`): direct repository access (not the discover-aware
  path under `/mcp-registry/*`).
- **Eval** (`/eval/runs`, `/eval/replay/{sessionId}`): see §7.10.
- **Hooks** (`/hooks`): `hook_mappings` CRUD.

### 5.12 Admin (`/api/v1/admin/*` — JWT)

`POST /tools/reindex` — re-embeds every enabled tool into
`agent_tool_embeddings`.

## 6. Agent runtime — one chat turn end-to-end

### 6.1 Entry: `ChatController.chat()`

- Validates request, sets SSE headers (`X-Accel-Buffering: no`,
  `Cache-Control: no-cache`).
- Creates `SseEmitter(-1L)` (no timeout — Spring MVC `request-timeout: -1`).
- Delegates to `ChatService.handleChatAsync(request, emitter, userId)`.

### 6.2 `ChatService.handleChatAsync()`

1. Wrap emitter in `SseEventSender`; attach lifecycle callbacks
   (completion/timeout/error → `session.cancel()`).
2. Validate attachments, get-or-create session via `AgentSessionManager`.
3. **`buildEffectiveState`** merges persisted `session_context_state` with
   request overrides (timezone, runtime mode, model, profile, embedding
   model).
4. Restore history from DB if in-memory session is fresh.
5. Send `connected` event.
6. **Workspace provision:** `SessionWorkspaceService.getOrCreate(sessionId)`
   creates `${java.io.tmpdir}/pods-agent-vfs/{sessionId}/.pods-agent/`;
   `WorkspaceSkillSyncService.sync(workspace)` mirrors enabled skill files
   under `.pods-agent/skills/{slug}/{file}`.
7. **`session.beginTurn()`** — generates a fresh `turnId` (UUID).
8. **Rolling summary pre-pass:** `ContextSummarizationService.maybeSummarize`
   may compact older messages; if it did, update `state.rollingSummary` and
   emit `summary.updated`.
9. Enrich the user message with attachment metadata; on first message,
   optionally generate a session title asynchronously.
10. Save the user message with `role=user, turn_id=null`.
11. Run the turn inside both `WorkspaceContextHolder` and
    `UserContextHolder` thread-locals so tools can see the workspace + user:
    `agentRuntimeService.runTurn(...)`.
12. Save the assistant message with `role=assistant, turn_id=<thisTurn>`,
    send `done`, persist `session_context_state` and `cost_usage`,
    `sender.complete()`.

### 6.3 `AgentRuntimeService.runTurn()`

- Normalise `runtimeMode` (default `aicore_loop`).
- Push a "bare" `UserMessage` for this turn into the in-memory session.
- If `modelSelectionMode == "auto"`, run `ModelAutoRouterService.pickModel`
  (complexity score by length/words/code tokens/history) → set `state.model`,
  emit `state.updated`.
- Fire `pre-prompt` hook via `RuntimeHookRegistryService`.
- Emit `plan.created`; transition planner FSM to `PLAN`; persist a
  `RuntimeTrace(traceType=plan, correlationId=state-PLAN)` plus optional
  `state.transition` `RuntimeEvent`.
- **Fast paths:**
  - Capabilities listing (the user is asking "what can you do?") →
    SYNTHESIZE → DONE.
  - Strict-scope pre-flight refusal (when no enabled skill/tool has cosine
    similarity ≥ `pods.runtime.strict-scope.min-top-tool-cosine`) → DONE
    with `pods.runtime.out-of-scope-refusal-message`.
- Otherwise → `runModelDrivenCoreLoop()`.

### 6.4 `runModelDrivenCoreLoop()`

1. Transition to PLAN; emit `task.started("MODEL_STEP")`.
2. If history is over budget, **compact in-loop** — same summariser, then
   `retainRecentMessages`, update rolling summary, emit `summary.updated`.
3. **`selectTurnTools(...)`** — see §6.5.
4. Skill-first gate check — see §6.6.
5. Build **step context**:
   - The user's literal text
   - Selected skill content (XML-tagged)
   - MCP hints
   - Workspace manifest (filtered to non-dotfiles, excluding `.pods-agent`)
   - Entity carry-forward from previous turn
   - VFS policy + runtime mode
6. `AgentToolCallbackFactory.buildForTurn(...)` materialises Spring AI
   `ToolCallback`s — wraps each `AgentTool` with `AgentToolCallback`, adds
   synthetic memory/retrieval/skill callbacks.
7. `AgentOrchestrator.streamTurn(...)` runs the model.

### 6.5 Tool selection (`selectTurnTools`)

1. `hardCap = pods.runtime.max-tool-callbacks-per-turn` (default 120).
2. `topK = min(hardCap, toolRetrieval.topK or 40)`.
3. **Semantic retrieval** via `ToolEmbeddingIndexService.searchTopK`,
   biased by:
   - **Base injected** tools (must always be candidates)
   - **Memory signals** — per-tool scores stored in the user's memory file
     `.system/selection-signals.json`, merged with the per-session map
   - **Host affinity** — tools whose `host` matches a host already invoked
     this session (scanned from recent `tool.call` `runtime_events`)
4. **`ov-*` profile constraint**: if `state.agentProfileId` starts with
   `ov-`, the tool set is restricted to
   `{ ovListRunsForOrder, ovGetRunDetail, ovStartValidation,
   ovDashboardStats, question }`. Otherwise those `ov*` tools are stripped
   from general chat.
5. `capToProviderLimit` always includes the full base-injected set first,
   then fills remaining slots from the retrieval list without duplicates.

### 6.6 Skill-first gate (`SkillExecutionGate`)

When enabled, `AgentToolCallback` blocks any non-`skill`/`skillsearch`/
`toolsearch` tool until the model has loaded a SKILL via the `skill` tool.
It returns `skill_first_required: ...` with hints. Enabled when:
- The current profile is **not** order-validation (`ov-*`), AND
- The heuristic `shouldEnforceSkillFirst(userText, sessionId)` matches —
  e.g. task-like utterances or a lexical match to an enabled skill.

This forces SKILL.md sequencing for skills that have hard rules (like
`pods-order-validation` which requires a precise tool-call order).

### 6.7 HITL: questions & approvals (`PendingInteractionService`)

- `create()` inserts a `hitl_interactions` row with `status=pending` plus a
  `CompletableFuture` in memory; supports `externalRequestId` to dedupe.
- `awaitReply(requestId, timeoutMs)` blocks the tool thread.
- The HTTP endpoints `POST /chat/reply | /chat/approve | /chat/reject`
  resolve the future and update the row.

**Question flow (actively used):** `AgentToolCallback` intercepts the
synthetic `question` tool. If the tool returns `approval_required:` it
calls `create(..., "question", prompt)`, emits the `question` SSE event,
persists a `question` `RuntimeEvent`, then `awaitReply`.

**Approval flow (wired but dormant):** The same callback handles
`GuardrailPolicyEngine.Decision.decision() == "ask"` → emits
`approval_required`, awaits reply. However, the engine currently rewrites
all `ask` decisions to `allow` (see §6.11), so this branch is effectively
disabled unless the engine is changed.

### 6.8 Model routing (`ModelProviderRouter`)

Resolves a `ModelRef` to a Spring AI `ChatClient`, reading the encrypted
key + base URL from `supported_models`. Provider IDs handled:

| ID | Backend |
|---|---|
| `anthropic` | Anthropic API (with optional `ThinkingConfigAdaptive` when `pods.runtime.enable-anthropic-thinking=true`) |
| `azure_claude` | Anthropic via Azure Foundry SDK |
| `azure` / `azure_openai` | Azure OpenAI |
| `openai` | OpenAI |
| `google` / `google-vertex` | OpenAI-compatible Gemini |
| `ollama` | Local Ollama |
| `default` | Any OpenAI-compatible base URL |

`ModelAutoRouterService.pickModel(...)` is called when
`modelSelectionMode=auto`. It scores complexity by length, code-ish tokens,
and history size, then picks among enabled `ModelConfig` rows by tier (cheap
vs strong) via name heuristics (`mini`/`haiku` vs `opus`/`gpt-4.1`).

### 6.9 Memory subsystem

- **Domain:** `memories` table — `(user_id, memory_file_path)` unique, with
  `category` ∈ `{user, feedback, project, reference}`, `tags TEXT[]`,
  `content`.
- **Service:** `MemoryService` — CRUD plus `buildInjectionPrompt(userId,
  rollingSummary, 2000)` which is appended to the system prompt for
  authenticated users.
- **Tools:** synthetic `memoryview`, `memorycreate`, `memorystrreplace`,
  `memoryinsert`, `memorydelete`, `memoryrename` — defined in
  `MemoryToolDefinitions`, dispatched via `ToolExecutionService.execute`
  branch → `MemoryTools` (which uses `UserContextHolder.currentUserId()`).
- **Selection signals:** Stored as a special memory row at
  `.system/selection-signals.json` with maps `tool`, `skill`, `toolDomain`
  → score (used by tool retrieval re-ranking).

### 6.10 Per-session workspace VFS

- Root: `${java.io.tmpdir}/pods-agent-vfs/{sessionId}/` with sub-directory
  `.pods-agent/`.
- Skill files mirrored into `.pods-agent/skills/{slug}/{file}` so the model
  can `read` them via the `read` tool without hitting the DB.
- Tool outputs above `pods.runtime.tool-output-vfs-spill-threshold-chars`
  (16 384) are written to
  `.pods-agent/turns/{turnId}/tool-results/{tool}-{callId}.json|txt`,
  replaced in the model's view by an envelope with `storedInVfs`, `path`,
  `preview`.
- `InstructionLoaderService` walks up from the workspace looking for
  `AGENTS.md` / `CLAUDE.md` — content goes into the system prompt under
  **"## Workspace Instructions"**.
- `ExecutionLogService` writes a per-turn structured trace under
  `.pods-agent/execution-logs/`.

### 6.11 Guardrails (`GuardrailPolicyEngine`)

Per-tool evaluation order:
1. Experimental tool + not whitelisted → **deny**.
2. `permissionScope` not in `pods.runtime.allowed-permission-scopes` → deny.
3. `requiresApproval=true` → **allow** with reason `auto_allow_requires_approval`.
4. DB `guardrail_policies` row with `scope=tool, rule_type=name,
   rule_value=<toolName>`: first match wins — but `ask` is rewritten to
   `allow` before returning.
5. Else: **allow** (`open_dev_default`).

Net effect: the engine **never returns `ask`** in current code. The
`approval_required` SSE/HITL plumbing exists end-to-end but is dormant.

### 6.12 Context summarization

`ContextSummarizationService.maybeSummarize(...)` compares estimated tokens
against `pods.runtime.summary-token-threshold` (6000). When over, it
produces a textual rolling summary, optionally merging with the previous
summary, and returns a `CompactionResult` with the number of removed
messages. Called both pre-turn in `ChatService` and in-loop in
`runModelDrivenCoreLoop`. Separately, `appendBudgetHint` adds a
**"## Context Budget"** stanza to the system prompt when estimated history
usage exceeds the configured warn/elevated/critical percentiles.

### 6.13 Planner FSM

`AgentRuntimeService.PlannerState`: `PLAN, APPROVAL, EXECUTE_PARALLEL,
SYNTHESIZE, REPLAN, DONE`. **Actually used** states: `PLAN` (turn start +
core loop), `SYNTHESIZE` (capabilities fast path), `DONE` (terminal).
`APPROVAL`, `EXECUTE_PARALLEL`, `REPLAN` are defined but unreferenced.

### 6.14 System prompt assembly (`AgentOrchestrator.buildSystemPrompt`)

If `state.agentProfileId` resolves to an `AgentProfile` with non-blank
`systemPrompt`, the orchestrator returns **profile text + `Current date:`
only** — no base prompt, retrieval contract, workspace blocks, rolling
summary, memory injection, or budget hints are appended. (The code comment
suggesting memory still appends is out of date.)

Otherwise the full assembly is:
1. `prompts/base-system-prompt.md` (or fallback string).
2. `Current date: <timezone>`.
3. **Retrieval catalog contract** explaining `skillsearch`, `toolsearch`,
   `skill`.
4. If a workspace exists: `## Workspace Instructions` (AGENTS.md / CLAUDE.md)
   + `## Workspace Skill Manifest` (`workspace://skills`).
5. If `state.rollingSummary` is set: `Conversation summary context:`.
6. If a user is authenticated: `AUTO_MEMORY_TOOLS_SYSTEM_PROMPT.md` +
   `MemoryService.buildInjectionPrompt(userId, rollingSummary, 2000)`.
7. Budget hint (`appendBudgetHint`).
8. Anthropic prompt-cache hint (`appendProviderCachingHint`).

## 7. Tools, skills, MCP, DMN

### 7.1 `AgentTool` (domain) — key columns

`id`, `domain_id`, `name`, `description`, `source_type`, `execution_kind`,
`permission_scope`, `requires_approval`, `model_gate`, `provider_gate`,
`experimental`, `input_schema_version`, HTTP fields (`method`, `host`,
`endpoint`, `request_schema`, `response_schema`), auth fields
(`auth_profile_id`, `auth_override_enabled`, `auth_type`, `auth_config`,
OAuth client/token URLs, `encrypted_*` tokens), `enabled`, `base_injected`.

`source_type` values in this repo:

- **`framework_default`** — seeded by `FrameworkToolPackService` at startup.
- **`openapi_import`** — OpenAPI import; **also** used for MCP-discovered
  tools (legacy naming).
- **`curl_import`** — cURL/Postman imports.
- **`manual`** — hand-authored via the UI.

`builtin` is also used in code-defined synthetic tools (memory, retrieval).

### 7.2 Tool execution (`ToolExecutionService.execute`)

`resolveExecutionKind(tool)` prefers `execution_kind`, else infers from
tool name. Branches:

| Kind | What runs |
|---|---|
| `filesystem` | `read`, `glob`, `grep`, `write`, `edit`, `apply_patch` — all paths clamped under `WorkspaceContextHolder.current()` workspace |
| `shell` | `zsh -lc` with **20s timeout**, cwd clamped to workspace |
| `web` | `webfetch` (plain GET) + `websearch`/`codesearch` via Exa MCP (JSON-RPC or SSE parsed) |
| `memory` | Delegates to `MemoryTools` with `UserContextHolder` (stubs for unimplemented `memory_save`/`memory_search`) |
| `integration` | `decisionTableEvaluate` → `DecisionTableService`; MCP proxy → `McpClientService.callTool`; `toolsearch`/`skillsearch` → `CatalogSearchService`; `question` → `approval_required:` envelope; stubs for `plan_exit`, `task`, `parallel_task`, `batch`, `pipeline`, `todowrite`, `agent_send`, `agent_receive`, `skill`, `lsp` |
| Default / `http_proxy` | JDK `HttpClient` GET/POST/PUT/PATCH/DELETE with path-param hydration, optional query mirroring, retries, per-tool circuit breaker, auth via `HttpToolAuthService` |

There is a separate `service/codeexec/CodeExecutionService` for sandboxed
code execution, governed by `pods.runtime.code-execution-*-timeout-ms` and
`*-memory-limit-mb`.

### 7.3 Framework default tools (`FrameworkToolPackService`)

Installed at startup into two domains, all with `source_type =
framework_default`:

**Framework Core Tools — base-injected (always in the turn's tool list)**

`question`, `plan_exit`, `todowrite`, all `memory*` tools,
`decisionTableEvaluate`, `toolsearch`, `skillsearch`.

**Framework Core Tools — retrieval-eligible (added when semantically relevant)**

`read`, `glob`, `grep`, `edit`, `write`, `apply_patch`, `bash`, `webfetch`,
`websearch`, `codesearch`, `task`, `skill`.

**Framework Experimental Tools** (`experimental = true`, gated by
`pods.runtime.enable-experimental-tools` + `enabled-experimental-tool-names`)

`lsp`, `batch`, `parallel_task`, `agent_send`, `agent_receive`, `pipeline`,
`memory_save`, `memory_search`.

### 7.4 Tool retrieval — `ToolEmbeddingIndexService`

- Builds embed text from name + description + domain + source + method/endpoint
  + truncated schemas.
- Embeds via the configured Spring AI `EmbeddingModel`, **pads/truncates to
  3072 floats** (`MAX_DIM`).
- Persisted as **`halfvec(3072)`** in `agent.agent_tool_embeddings`,
  indexed with **HNSW (`halfvec_cosine_ops`)**.
- Search: `1 - (embedding <=> :query)` with filters on
  `(model_provider, model_id, dimensions)`, fetches `topK*2`, then re-ranks
  in Java with memory signals (`tanh * weight`), host affinity (+0.25), and
  a configurable score floor.
- Sync triggers: `ToolRegistryService.refresh()` upserts by content hash;
  full re-index via `POST /api/v1/admin/tools/reindex`.

### 7.5 Skills (`Skill`, `SkillFile`)

- DB stores skill metadata + per-file metadata; **file content lives in
  Azure Blob** (via `SkillFileStorageService` → `AzureBlobSkillFileStorageService`).
- `SkillStorageConfig` falls back to `InMemorySkillFileStorageService` when
  no Azure config and `allow-in-memory-fallback=true`.
- `SkillRegistryService.refresh()` builds `SkillSnapshot(skill, Map<path,
  content>)` for enabled skills.
- **Default classpath skills** loaded from `default-skills/*.md` and
  `default-skills/**/*` directory bundles, with YAML frontmatter; IDs
  prefixed `system-`. **Defaults take precedence** over DB skills with the
  same name.
- `WorkspaceSkillSyncService.sync(workspace)` writes the enabled set to
  `.pods-agent/skills/{slug}/{file}` so tools can read them via `read`/`grep`.
- The `skill` tool is implemented by `SkillToolCallback` (not the
  `ToolExecutionService` stub), and returns a `<skill_content>` XML block to
  the model. Loading a skill flips `SkillExecutionGate.skillLoaded = true`,
  unblocking other tools.

### 7.6 Default skill bundled in this repo

Only **`pods-order-validation`** ships in `src/main/resources/default-skills/`
(`SKILL.md` + `templates/validate-order-workflow.json`). It enforces the
strict tool-call sequence described in §8.

### 7.7 MCP integration (`McpClientService`, `McpAuthService`)

- **Transport:** JSON-RPC POST over HTTP; tries candidate URLs derived from
  `base_url` + `streamable_path`, `sse_path`, `/mcp`, `/sse`,
  `/streamable`, `/api/mcp`, `/` until one stops 404-ing.
- **Body type:** accepts `application/json` or `text/event-stream`
  (last `data: {...}` line is parsed).
- **Discovery:** `tools/list` → caches normalised list with per-tool
  `enabled` flags in `mcp_registry.discovered_tools_json`.
- **Execution:** `tools/call` with arguments parsed from JSON `userText` or
  `{ query: raw }`. GitHub-specific shortcuts exist for repo listing and
  query normalisation.
- **Sync to `agent_tools`:** registers each discovered tool under domain
  `MCP {serverName}` with local name `mcp_{sanitized_remote_name}`,
  `source_type=openapi_import`, `execution_kind=integration`, and a
  `request_schema` JSON wrapper `{ mcpServerId, mcpToolName, inputSchema,
  headers }`. Stale `mcp_*` tools are disabled when discovery no longer
  lists them.
- **Auth modes** (`McpAuthService`): `api_key_header`, `bearer_token`,
  `basic_auth`, `oauth_client_credentials`, `oauth_auth_code` (PKCE S256),
  plus dynamic client registration to the standard `/register` endpoint.

### 7.8 HTTP tool auth (`HttpToolAuthService`)

Resolution order for an HTTP-proxy tool call:
1. If `auth_override_enabled` is true and `tool.auth_type` is set, use the
   credentials stored on the tool row (`AuthState.forTool`).
2. Else, if **exactly one enabled `tool_auth_profile`** exists for
   `tool.domain_id`, use it.
3. Else, no auth.

Supported auth kinds: `none`, `api_key_header`, `api_key_query`,
`bearer_token`, `basic_auth`, `oauth_client_credentials`, `oauth_auth_code`,
`custom_token_api` (templated POST/GET token endpoint with JSON-path for
token + expiry).

### 7.9 Decision tables (DMN)

- **Storage:** `decision_tables(name UNIQUE, dmn_json, hit_policy,
  metadata_json)`.
- **Parser:** `DmnJsonParser` expects `{hitPolicy, inputs[], outputs[],
  rules[{inputs, inputEntries, outputs, outputEntries}]}`.
- **Hit policies:** `FIRST`, `UNIQUE` (treated like FIRST: first match
  wins), `COLLECT` (merge outputs from every matching rule).
- **FEEL subset (`dmn/feel/FeelExpressionEvaluator`):** wildcard `-`/`*`,
  numeric comparisons `< <= > >=`, numeric ranges `[a..b]` / `(a..b)` with
  inclusive/exclusive endpoints, quoted strings, boolean equality, numeric
  and string equality. **Not full FEEL** — no function calls, no
  arithmetic, no temporal types.
- **Tool integration:** the framework tool `decisionTableEvaluate` is wired
  in `ToolExecutionService.executeIntegration` and required for the order
  validation flow.

### 7.10 Eval harness (`EvalHarnessService`)

Currently a **stub**: `launch` inserts an `eval_runs` row, then `@Async
runAsync` sleeps `max(750, datasetSize * 150)`ms and writes a deterministic
pseudo-score derived from a dataset hash. No real model/tool execution.
`replaySessionTrace` aggregates `runtime_traces` for a session into a
deterministic hash + item list — useful as a baseline for trace diffs.

## 8. The PODS order-validation flow

This is the canonical business workflow this agent was built to run end-to-end.

The skill `pods-order-validation/SKILL.md` (loaded automatically as a
classpath default) defines a **strict, no-fabrication, no-batching** pipeline.
The model is told to keep calling tools until a checklist is satisfied — it
must not summarise early.

**Required resources (registered out-of-band as tools + a decision table):**

| Resource | Type | Frequency |
|---|---|---|
| `Get_OrderID` | HTTP tool | once per turn |
| `Leg Sequences` | DMN decision table | once, via `decisionTableEvaluate` |
| `Serviceability` | HTTP tool | once per non-skipped leg line |
| `ContainerAvailability` | HTTP tool | once per qualifying IDEL line |

**Flow:**

1. Call `Get_OrderID` with `ORD_ID`.
2. Filter `order.Lines` by `ItemCode` ∈ a fixed allow-list (`IDEL`, `RETSC`,
   `LDT`, `REDEL`, `FPU`, `MOV`, `SID`, `SFP`, `SCDEL`, `CSRED`, `CSFPU`).
3. Map ItemCode → ServiceCode (the decision table expects service-code
   form). Call `decisionTableEvaluate(tableName="Leg Sequences",
   inputs={actualSequence:[...]})`. A `{matched:false}` result is reported
   verbatim — no retries.
4. For each leg line, call `Serviceability`. Interpretation rule: only a
   `CallInfo.StatusCode` containing `"OutOfMarket"` makes the line fail.
   `PostalCodeException` and other warnings are informational and still
   pass.
5. For each `IDEL` line that has no `DeliveryDate` and either no
   `ContainerId` or no `ScheduledDate`, call `ContainerAvailability`. Collect
   dates where `IsAvailableCS=true` and `ReasonCode` is empty.
6. Final answer is a strict JSON object with `legSequence`, `serviceability`
   (one entry per leg, including skipped), and `containerAvailability`.

The runtime enforces this by:
- Auto-loading the skill when the user prompts an OV intent (via
  `agentProfileId` starting with `ov-` or via skill semantic match).
- Restricting the tool set for `ov-*` profiles to the OV tools plus
  `question`.
- The skill's own hard-rule prose telling the LLM to never stop early.

### 9.2 Repository pattern

- All repositories use `JdbcTemplate` / `NamedParameterJdbcTemplate`.
- `SqlQueryLoader` loads `sql.properties` from the classpath once at
  construction; repositories fetch SQL strings by key.
- Three shared row mappers under `repository/mapper/`:
  `ChatSessionRowMapper`, `ChatMessageRowMapper`, `ModelConfigRowMapper`.
  Other repositories use inline lambdas.
- Two repositories use inline SQL (not externalised): `ToolAuthProfileRepository`,
  and `AgentToolRepository.updateAuthBinding(...)`.

### 9.3 pgvector specifics

```sql
CREATE EXTENSION IF NOT EXISTS vector;
-- column starts as vector(3072), then:
ALTER TABLE agent.agent_tool_embeddings
  ALTER COLUMN embedding TYPE halfvec(3072);
-- HNSW index:
CREATE INDEX idx_tool_embeddings_hnsw
  ON agent.agent_tool_embeddings
  USING hnsw (embedding halfvec_cosine_ops);
```

Search uses cosine distance (`<=>`) with similarity `= 1 - distance`. The
service inserts via `CAST(:embedding AS vector)` and pgvector handles the
conversion to halfvec on storage.

### 9.4 Sensitive columns

Encrypted with **AES-256-GCM** (`EncryptionService`, key from
`pods.encryption.key` / env `PODS_ENCRYPTION_KEY`, format
`base64(iv):base64(tag):base64(ciphertext)`):

- `supported_models.encrypted_key`
- `agent_tools.encrypted_client_secret / encrypted_access_token / encrypted_refresh_token`
- `tool_auth_profiles.encrypted_client_secret / encrypted_access_token / encrypted_refresh_token`
- `mcp_registry.encrypted_client_secret / encrypted_access_token / encrypted_refresh_token`

Hashed (BCrypt, separate path): `users.password_hash`.

### 9.5 One turn's DB footprint

For a single chat turn that uses one tool, the rows touched are typically:

- `chat_sessions`: `last_active` updated.
- `session_context_state`: upserted at end with `runtime_mode`,
  `model_ref`, `rolling_summary`, `summary_tokens`, `state_json`.
- `chat_messages`: user row (`turn_id=null`) + assistant row (`turn_id=<this>`).
- `runtime_events`: `plan.created`, `tool.call`, `tool.done`, `task.done`
  (also `reasoning` if non-blank, plus optional `state.transition`,
  `workspace.ready`, `skills.synced`).
- `runtime_traces`: one row at PLAN transition (`trace_type=plan`).
- `cost_usage`: one row with `turn_id` and provider/model/token counts.
- `hitl_interactions`: only if a `question` or `approval_required` event
  was raised.

`truncateSessionFromMessage` cascades correctly — it calls `deleteFromTime`
on `runtime_events`, `runtime_traces`, `hitl_interactions`, `cost_usage`
using the truncated message's `created_at`.

## 13. Risks, dead code, and recommendations

### 13.1 Security findings

- **Hard-coded production-looking credentials in `application.yaml`:**
  - `spring.datasource.password: npg_u8zSPQ2TtYpw` (Neon DB)
  - `pods.skills.blob.connection-string` includes an Azure Storage AccountKey.
  - `pods.auth.jwt-secret` and `pods.encryption.key` are checked in.
  
  These should be moved to environment variables / a secret store before
  any non-dev deployment. The JWT secret in particular signs every user
  token — leaking it allows arbitrary impersonation.

- **`/api/v1/models` POST is `permitAll`.** Anyone reachable on the API
  can register a model with arbitrary `base_url` + `apiKey`, which can be
  used to exfiltrate prompts (point the agent at an attacker-controlled
  OpenAI-compatible URL). Same applies to `/api/v1/chat` itself which is
  also `permitAll` but uses a default user id.

- **`order-validation-ui` default user:** unauthenticated chat requests
  run as a fixed user id. Any data scoped to that user (memories, sessions,
  cost) is shared across all unauthenticated callers.

- **`GuardrailPolicyEngine` rewrites `ask` → `allow`.** The interactive
  approval path is plumbed end-to-end but currently disabled, including
  for tools with `requires_approval = true`. If this is intentional for
  dev, document it; otherwise the engine should respect policy decisions.

- **No CSRF protection** (disabled in `SecurityConfig`). Acceptable for a
  pure API + bearer-token frontend, but worth noting.

### 13.2 Dead code

- **Empty UI dirs:** `UI/src/components/toolchain/` and
  `UI/src/components/workflow/` (and its `designer/`, `inspector/`, `run/`
  sub-folders) — leftover from the toolchain removal.
- **Orphaned UI:** `components/decision/RulesGrid`,
  `DecisionTableTestPanel`, `types/decision.ts`, and the entire
  `api.decisionTables.*` namespace — not imported by any route/page.
- **Orphaned UI libs:** `UI/src/lib/workflowLayout.ts`,
  `workflowAutoLayout.ts`, `chatStream.ts` — only reference each other.
- **Unused npm deps:** `zustand`, `react-hook-form`, `@google/genai`,
  `@monaco-editor/react`, `@xyflow/react`, `motion`, `recharts`,
  `@dagrejs/dagre`.
- **Orphaned tests:** the entire `src/test/java/com/pods/agent/workflow/`
  package + `WorkflowProposalAsyncServiceTest` reference deleted main
  classes — they will fail to compile.
- **`ArchitectNoteCallback`** is defined under `agent/tool/` but is **not
  registered** in `AgentToolCallbackFactory.buildForTurn`. It is unreachable.
- **Planner states `APPROVAL`, `EXECUTE_PARALLEL`, `REPLAN`** are defined
  but never used.
- **Unused entries in `pods-order-validation/` root** — duplicates of the
  classpath skill copy; the runtime loads `src/main/resources/default-skills/`.

### 13.3 Code-comment / doc drift

- `AgentOrchestrator.buildSystemPrompt` has comments suggesting memory
  tools still append when a custom profile prompt is used, but the early
  `return` skips that — comments are out of sync with code.
- Skill `pods-order-validation` lists `Get_OrderID`, `Serviceability`,
  `ContainerAvailability` as required tools, but none of them are seeded
  by `FrameworkToolPackService` — they have to be registered manually via
  the Tools UI for the flow to work end-to-end.
- The MCP-synced tools are stored with `source_type=openapi_import`
  (legacy). The UI hides them with a `mcp` host check, but the column
  value is misleading. Consider a dedicated `source_type=mcp_discovered`.

### 13.4 Recommended cleanup (low risk)

1. Delete `UI/src/components/toolchain/` and `UI/src/components/workflow/`
   (empty directories left from prior removal).
2. Delete `src/test/java/com/pods/agent/workflow/*` and
   `WorkflowProposalAsyncServiceTest.java`.
3. Decide on `components/decision/*` + `api.decisionTables.*` — either
   wire a `DecisionTablesPage` route or delete.
4. Remove unused npm deps (`zustand`, `react-hook-form`, `@xyflow/react`,
   `@monaco-editor/react`, `@google/genai`, `motion`, `recharts`,
   `@dagrejs/dagre`).
5. Delete `UI/src/lib/{workflowLayout,workflowAutoLayout,chatStream}.ts`
   (unreferenced).
6. Decide whether `ArchitectNoteCallback` should be registered or removed.
7. Trim unused `PlannerState` enum values.

### 13.5 Recommended hardening

1. Move `spring.datasource.password`, `pods.auth.jwt-secret`,
   `pods.encryption.key`, and `pods.skills.blob.connection-string` to env
   vars and rotate the leaked values.
2. Decide which `/api/v1/models/**` operations should require auth.
3. Audit `permitAll` paths in `SecurityConfig` for production.
4. Either restore the `ask`-decision path in `GuardrailPolicyEngine` or
   document that all approval gates are currently bypassed in dev.

## 14. How to run locally

### 14.1 Prerequisites

- Java 25
- Maven 3.9+
- Postgres 14+ with the `vector` extension installed (or pgvector-enabled
  Neon)
- Node 20+ (for the UI)

### 14.2 Database

`src/main/resources/db/model.sql` is idempotent — apply it manually to your
Postgres instance:

```bash
psql "<your connection string>" -f src/main/resources/db/model.sql
```

It creates the `agent` schema and all tables, plus the
`agent_tool_embeddings` HNSW index. Re-running it is safe.

### 14.3 Backend

Set environment variables to override the dev defaults in
`application.yaml` (strongly recommended — the file has live credentials):

```bash
export SPRING_DATASOURCE_URL='jdbc:postgresql://<host>:5432/ai_agent'
export SPRING_DATASOURCE_USERNAME=<user>
export SPRING_DATASOURCE_PASSWORD=<pwd>
export PODS_AUTH_JWT_SECRET='<change-me-256-bit>'
export PODS_ENCRYPTION_KEY='<64-hex-aes-key>'
# Optional:
export PODS_SKILLS_BLOB_CONNECTION_STRING='DefaultEndpointsProtocol=...'
```

Run:

```bash
mvn spring-boot:run
# or: ./mvnw spring-boot:run
```

The service listens on `:8080` and exposes Swagger UI at `/swagger-ui.html`
(via springdoc).

On startup, `FrameworkDefaultToolsBootstrap` seeds all framework tools and
`ToolRegistryService.refresh()` populates the in-memory cache and the
embedding index.

### 14.4 Frontend

```bash
cd UI
npm install
npm run dev
```

By default the UI calls `http://localhost:8080/api/v1/*`. Open
`http://localhost:5173/`, sign up, then visit `/chat`.

### 14.5 First-time setup checklist

1. Open `/models` — register at least one chat model with an API key.
2. Open `/embedding-models` — register one embedding model and mark it
   default. Click "Reindex tools" (or POST `/api/v1/admin/tools/reindex`)
   so semantic tool retrieval works.
3. Open `/skills` — confirm `pods-order-validation` is listed as a
   classpath default; upload/enable any additional skills you need.
4. Open `/tools` — for OV flows, create a domain, import or hand-author
   the three OV tools (`Get_OrderID`, `Serviceability`,
   `ContainerAvailability`), and create a domain-level auth profile.
5. Open `/chat` — pick a model and start a turn. Use
   `agentProfileId=ov-*` (or just ask in natural language) to invoke the
   OV flow.

## 15. Glossary

| Term | Meaning |
|---|---|
| **Agent profile** | A row in `agent_profiles` that pins a system prompt + runtime mode for a session. `ov-*` profiles restrict the tool set to OV tools. |
| **Skill** | A bundle of markdown + supporting files that the LLM can load via the `skill` tool. Default skills ship on the classpath; user skills live in Postgres + Azure Blob. |
| **Base-injected tool** | A framework tool that is **always** included in the per-turn tool list (memory, retrieval, decision-table, question, plan-exit, todowrite). |
| **Workspace VFS** | The per-session temp directory at `$TMPDIR/pods-agent-vfs/{sessionId}/` that holds synced skill files and tool output spills. |
| **Step context** | The user-facing message Spring AI receives for the current turn — includes the user's text plus skill content, MCP hints, workspace manifest, and entity hints. |
| **Turn id** | UUID stamped on all messages, runtime_events, runtime_traces, cost_usage, and hitl_interactions belonging to one user → assistant exchange. |
| **HITL** | Human-in-the-loop — when the agent asks the user a question or requests approval mid-turn, blocking on `PendingInteractionService.awaitReply`. |
| **MCP** | Model Context Protocol — external tool servers federated via JSON-RPC. |
| **DMN** | Decision Model and Notation — the format for decision tables (subset implemented in `dmn/`). |
| **FEEL** | The expression language used inside DMN tables (subset implemented in `dmn/feel/`). |
